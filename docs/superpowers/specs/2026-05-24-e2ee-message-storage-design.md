# 端到端加密消息存储设计

## 背景
当前聊天消息以明文形式保存在数据库中，服务端在消息创建、历史查询、WebSocket 转发链路中都能看到消息正文。新的目标是把聊天消息改为真正的端到端加密（E2EE）：消息在发送端加密，服务端只保存和转发密文，接收端本地解密后展示。

## 本次确认的范围
- 采用真 E2EE，而不是仅数据库加密
- 密钥对由前端浏览器本地生成
- 私钥只保存在本地，不上传服务端
- 服务端只保存用户公钥和消息密文
- 不支持私钥恢复
- 用户换设备或清缓存后，旧消息正文不可恢复
- 如果接收方没有公钥，则不允许发送消息
- 用户将直接删除并重建数据库，因此 `init.sql` 直接更新为新表结构，不做旧库迁移兼容

## 总体架构
### 发送端
1. 用户登录后，前端检查本地是否已有当前账号的密钥对。
2. 若不存在，则前端使用浏览器原生 Web Crypto API 生成密钥对。
3. 前端将公钥上传到服务端，私钥只保存在浏览器本地。
4. 发消息前，前端获取接收方公钥。
5. 前端用接收方公钥加密消息明文，得到密文后再调用发送接口。

### 服务端
1. 保存用户公钥。
2. 发送接口只接收密文，不接收明文消息内容。
3. 数据库只保存密文消息。
4. WebSocket 只转发密文，不参与解密。
5. 历史消息接口只返回密文和展示所需元数据。

### 接收端
1. 前端收到历史消息或 WebSocket 实时消息后，使用本地私钥解密。
2. 解密成功则显示明文。
3. 解密失败、无私钥、私钥已更换时，消息正文显示为 `***（已加密）`。
4. 即使无法解密，也要像正常消息一样显示发送时间、左右气泡和发送者身份。

## 数据模型设计
### 用户公钥
用户记录需要补充公钥相关字段，最小集合如下：
- `public_key`：用户当前公钥
- `public_key_algorithm`：公钥算法标识，例如 `rsa-oaep-sha256`
- 可选 `key_version`：后续若支持密钥轮换可用；MVP 阶段可先固定为 1

### 消息表
消息表不再保存明文 `content`，改为保存密文语义明确的字段：
- `sender_id`
- `receiver_id`
- `ciphertext`：密文正文
- `algorithm`：消息加密算法标识
- `created_at`
- 可选 `iv` / `nonce`：如果所选浏览器加密实现需要额外参数则一并保存
- 可选 `key_version`：若后续支持密钥轮换可加上；MVP 阶段可以先固定值

设计原则：避免继续使用 `content` 这种容易被误认为明文的字段名，防止后续误用。

## API 设计
### 上传或更新我的公钥
`PUT /api/users/me/public-key`

请求体：
```json
{
  "publicKey": "-----BEGIN PUBLIC KEY-----...",
  "algorithm": "rsa-oaep-sha256"
}
```

用途：
- 首次生成密钥对后上传公钥
- 本地私钥丢失后重新生成密钥对并上传新的公钥

### 获取好友公钥
推荐把公钥并入现有 `GET /api/friends` 返回结构，而不是单独增加查公钥接口。

返回中新增字段：
- `publicKey`
- `publicKeyAlgorithm`

这样聊天页拿好友列表时即可同时拿到聊天所需公钥，减少额外请求。

### 发送消息
现有发送消息接口改为仅接收密文：

`POST /api/messages`

请求体：
```json
{
  "receiverId": 123,
  "ciphertext": "base64-encoded-ciphertext",
  "algorithm": "rsa-oaep-sha256"
}
```

服务端校验：
- 双方必须是好友
- 接收方必须已经上传公钥
- 不接受明文 `content`

### 历史消息
`GET /api/messages?friendId=...`

返回每条消息的核心字段：
```json
{
  "id": 1,
  "senderId": 100,
  "receiverId": 200,
  "ciphertext": "base64-encoded-ciphertext",
  "algorithm": "rsa-oaep-sha256",
  "createdAt": "2026-05-24T09:05:00+08:00"
}
```

### WebSocket 实时消息
`chat_message` 的 `data` 结构也改为密文结构，不再带明文正文。这样：
- 发送成功响应
- 历史消息返回
- WebSocket 实时消息

三条链路的消息结构保持一致，前端只维护一套解密与回退展示逻辑。

## 前端交互规则
### 登录后
- 检查当前账号本地是否已有私钥
- 没有则生成新密钥对
- 将公钥上传服务端
- 私钥仅保存在浏览器本地

### 打开聊天页
- 从好友列表读取对方公钥
- 若对方没有公钥，则禁止发送，并提示“对方未启用端到端加密消息”

### 发送消息
- 前端先使用对方公钥加密明文
- 加密成功后才调用发送接口
- 加密失败则不发送，并给出错误提示

### 展示消息
- 解密成功：显示明文
- 解密失败或没有私钥：显示 `***（已加密）`
- 无论是否解密成功，时间格式继续遵循现有规则：
  - 当天消息：`HH:mm`
  - 跨天消息：`MM-DD HH:mm`

## 异常处理
### 没有本地私钥
- 历史消息正文显示 `***（已加密）`
- 时间正常显示
- 当前设备不允许发送新消息，并提示缺少私钥

### 接收方没有公钥
- 不允许发送
- 前端提示：`对方未启用端到端加密消息`

### 解密失败
可能原因包括：
- 本地私钥丢失
- 当前私钥不是生成该消息时对应的旧私钥
- 密文损坏

处理规则：
- 页面不能因为单条解密失败而中断
- 单条消息正文显示 `***（已加密）`
- 消息时间和外层 UI 继续正常展示

### 清缓存或换设备
- 新设备生成新密钥对并上传新公钥
- 之后的新消息可以继续发送和接收
- 旧消息正文大概率无法解密，统一显示 `***（已加密）`

## 安全边界
这次实现属于单设备、无恢复能力的 MVP 版 E2EE，边界如下：
- 服务端看不到消息明文
- 数据库存的不是消息明文
- 服务端保存用户公钥
- 私钥只存在浏览器本地
- 不支持私钥恢复
- 不支持多端同步解密
- 丢失本地私钥后，旧消息正文不可恢复

## 测试范围
### 后端
- 上传公钥接口可用
- 好友列表可返回公钥字段
- 发送消息接口拒绝明文消息体
- 发送消息接口只保存密文
- WebSocket 推送的是密文而不是明文
- 历史消息接口返回密文和时间，不返回明文

### 前端
- 首次登录能自动生成密钥对
- 公钥能成功上传
- 能使用好友公钥加密发送
- 收到密文后能成功解密展示
- 没有私钥时显示 `***（已加密）`
- 解密失败时显示 `***（已加密）`
- 时间格式保持现有规则

### 端到端
- A 发给 B，数据库里应为密文
- B 页面能正常解密显示
- 删除 B 本地私钥后再次打开历史消息，应显示 `***（已加密）`
- 该情况下消息时间仍正常显示

## 直接影响
由于用户决定删除并重建数据库，本次实现可以直接更新 `init.sql`：
- 用户表增加公钥相关字段
- 消息表移除明文内容字段，改为密文字段
- 不需要为旧表设计在线迁移或兼容写法

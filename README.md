# 基于前后端分离架构的即时通讯系统

## 一、项目简介
本项目是一个面向毕业设计场景实现的即时通讯系统，采用前后端分离架构进行开发，完成了用户注册登录、个人资料维护、好友申请与管理、实时聊天以及消息加密等核心功能。

系统前端基于 Vue 3 构建交互界面，后端基于 Go 语言实现业务服务，并通过 HTTP 接口与 WebSocket 长连接共同完成用户认证、好友关系维护与消息实时投递。项目在满足即时通讯基础能力的同时，重点围绕聊天消息安全性进行了设计与实现，使其具备较强的工程实践与展示价值。

## 二、功能概述
本项目当前已实现的主要功能如下：

- 用户注册与登录
- 用户个人资料查看与修改
- 好友申请发送、接收、同意与拒绝
- 好友列表展示
- 基于 WebSocket 的实时聊天通信
- 历史消息查询
- 基于本地密钥的消息加密发送与密文存储

- `init.sql`：数据库初始化脚本，用于创建项目所需表结构

## 四、消息加密设计与实现亮点
### 1. 设计目标
在即时通讯场景中，聊天内容通常具有较强的隐私属性。为提升消息传输与存储过程中的安全性，本项目在聊天模块中引入了基于非对称加密的消息保护机制，使服务端在多数场景下仅处理密文，而不直接接触消息明文。

### 2. 实现思路
本项目的消息加密机制采用“前端本地生成密钥 + 前端加密消息 + 服务端转发与存储密文”的设计思路，核心流程如下：

1. 用户首次进入聊天相关功能时，前端通过浏览器 `Web Crypto API` 生成 RSA 密钥对
2. 私钥保存在当前用户本地浏览器存储中，不上传至服务端
3. 公钥通过接口上传至服务端，用于后续好友之间加密通信
4. 用户发送消息时，前端分别使用发送者公钥和接收者公钥对同一条消息各加密一份
5. 服务端只接收、存储并转发密文数据
6. 聊天双方在本地使用各自私钥对对应密文进行解密显示

### 3. 当前方案的具体特点
与只在服务端保存明文消息的普通聊天系统相比，本项目方案具有以下特点：

- **私钥不上传服务端**：私钥仅保存在本地设备，有助于降低明文泄露风险
- **消息发送前完成加密**：聊天内容在浏览器侧完成加密后再提交给后端
- **服务端存储密文**：数据库中保存的是发送者密文与接收者密文，而非原始明文
- **兼顾消息接收与历史查看**：系统为发送者和接收者分别保存一份可由各自私钥解密的密文，使双方都可以查看本地聊天记录
- **提升项目展示亮点**：该设计不仅增强了系统安全性，也体现了项目在通信安全方向上的工程探索

### 4. 安全性说明
本项目已经具备端到端加密的基本特征，即消息明文主要存在于通信两端的浏览器环境中，服务端承担的是公钥保存、密文存储与消息转发职责，不直接参与消息明文处理。

但从严格意义上看，当前实现仍更接近“前端本地加密 + 服务端密文转发与存储”的方案，尚未达到工业级完整端到端加密体系。其主要原因包括：

- 当前尚未建立完善的公钥真实性校验机制
- 尚未实现前向保密能力
- 尚未引入会话密钥协商机制
- 尚未支持多设备密钥同步与设备级身份管理

因此，本项目的消息加密设计更适合作为毕业设计中的安全增强方案与工程实践成果展示，而不是直接等同于成熟商业即时通讯软件中的完整 E2EE 体系。

## 五、项目亮点与特色
除基础聊天功能外，本项目主要亮点如下：

1. **前后端分离架构清晰**  
   前端与后端职责划分明确，便于模块化开发、联调与后续扩展。

   从用户注册登录、资料维护、好友申请到实时聊天，形成了较完整的即时通讯业务闭环。

3. **引入消息加密机制**  
   在一般课程项目常见功能基础上，增加了消息加密能力，提升了系统安全性与项目创新性。

4. **实时通信能力较强**  
   通过 WebSocket 建立长连接，实现聊天消息的实时投递，增强了系统交互体验。

5. **具备一定安全控制意识**  
   系统结合 JWT 鉴权、好友关系校验与密文存储机制，在身份认证与消息安全方面具备较好的实践意义。

## 七、系统运行环境
### 1. 基础环境要求
- Node.js 18 及以上
- npm
- Go 1.22
- MySQL 8.x

### 2. 前端依赖安装
在 `web/` 目录下安装依赖：

```bash
cd web
npm install
```

### 3. 后端依赖说明
后端基于 Go Modules 管理依赖，进入 `server/` 目录后执行 `go run` 或 `go test` 时会自动解析依赖。

## 八、快速启动方式
### 1. 初始化数据库
确保本地已启动 MySQL 服务，然后执行根目录下的初始化脚本：

```bash
mysql -u root -p < init.sql
```

该脚本会创建 `whr_im` 数据库及 `users`、`friend_requests`、`friends`、`messages` 等核心表。

### 2. 修改后端配置
打开配置文件：

- `server/config/config.yaml`

根据本地 MySQL 环境修改数据库连接信息，例如当前配置格式为：

```yaml
server:
  port: ":8080"
mysql:
  dsn: "root:你的密码@tcp(localhost:3306)/whr_im?charset=utf8mb4&parseTime=True&loc=Local"
```

### 3. 启动后端服务
在 `server/` 目录执行：

```bash
cd server
go run ./cmd/main.go
```

后端默认监听：

```text
http://127.0.0.1:8080
```

健康检查接口为：

```text
GET /health
```

### 4. 启动前端项目
在 `web/` 目录执行：

```bash
cd web
npm run dev
```

前端开发环境默认通过 `web/.env.development` 中的配置访问后端：

```env
VITE_API_BASE_URL=http://127.0.0.1:8080/api
VITE_WS_BASE_URL=ws://127.0.0.1:8080/ws
```

如后端地址或端口变化，可同步调整上述环境变量。

## 九、系统使用流程
系统的基本使用流程如下：

1. 启动前后端服务
2. 访问前端页面，完成用户注册与登录
3. 在个人资料页完善昵称、性别、个性签名等信息
4. 通过用户名向其他用户发送好友申请
5. 在好友申请页面接收并处理好友请求
6. 好友关系建立后，进入聊天页面进行实时通信
7. 发送消息时，前端会在本地完成消息加密，再将密文提交到服务端
8. 接收方收到密文后，在本地完成解密并显示聊天内容

## 十、主要接口说明
系统当前核心接口包括：

### 1. 认证相关
- `POST /api/auth/register`：用户注册
- `POST /api/auth/login`：用户登录
- `GET /api/users/me`：获取当前用户信息
- `PUT /api/users/me`：更新当前用户资料
- `PUT /api/users/me/public-key`：上传当前用户公钥

### 2. 好友相关
- `POST /api/friend-requests`：发送好友申请
- `GET /api/friend-requests/incoming`：获取收到的好友申请
- `PUT /api/friend-requests/:id/accept`：同意好友申请
- `PUT /api/friend-requests/:id/reject`：拒绝好友申请
- `GET /api/friends`：获取好友列表

### 3. 消息相关
- `POST /api/messages`：发送消息（提交密文）
- `GET /api/messages?friendId=xxx`：查询与指定好友的历史消息
- `GET /ws?token=xxx`：建立 WebSocket 连接并接收实时聊天消息
